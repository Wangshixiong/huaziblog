# 发文手册

## 1. 适用范围

这份手册只解决一件事：

- 华哥以后如何在这个博客里新增、修改、检查并发布文章

它不负责部署、域名、Vercel、评论系统。这些内容在 `DEPLOY_GUIDE.md`。

## 2. 文章放在哪里

中文文章统一放在：

- `src/pages/posts`

每篇文章就是一个 Markdown 文件。

## 3. 文件命名规则

建议文件名格式：

- `57-你的标题.md`

规则说明：

- 前面的数字是期数
- 中间用连字符 `-`
- 后面是文章标题
- 文件名尽量不要再频繁改动，避免后续路由和 git 历史变乱

示例：

- `57-我最近在用的AI工具.md`
- `58-Dify工作流踩坑记录.md`

## 4. 最推荐的写法

建议每篇文章都包含完整 frontmatter。

模板如下：

```md
---
date: 2026/04/08
issueTitle: "这里写标题"
description: "这里写摘要"
image: "https://你的封面图地址"
heroImage: "https://你的封面图地址"
tags:
  - "AI"
  - "产品"
categories:
  - "博客"
---

<img src="https://你的封面图地址" width="800" />

正文从这里开始。
```

## 5. 每个字段怎么理解

### 5.1 `date`

文章发布日期。

建议格式：

- `2026/04/08`

不要混着写不同格式，后面排序会更稳。

### 5.2 `issueTitle`

文章标题。

这是站点真正用来显示的标题字段，建议和你希望读者看到的标题完全一致。

### 5.3 `description`

文章摘要。

这个字段会影响：

- 首页卡片摘要
- SEO 描述
- 分享时的摘要信息

建议写法：

- 1 到 2 句话
- 直接说明这篇文章讲什么
- 不要太长

### 5.4 `image` 和 `heroImage`

当前建议先保持一致：

- `image`
- `heroImage`

都写同一张封面图地址。

原因：

- 这样最稳
- 当前项目里没有必要把封面和头图拆成两套逻辑

### 5.5 `tags`

用于标识主题。

建议写你真正长期会重复使用的标签，比如：

- `AI`
- `Dify`
- `保险`
- `产品`
- `Agent`

### 5.6 `categories`

用于更粗粒度分类。

建议不要分太细，不然后面会越来越乱。

## 6. 封面图怎么处理

### 6.1 当前最稳方案

直接使用稳定可访问的 OSS 图片地址。

要求：

- 图片地址能被浏览器直接访问
- OSS 防盗链白名单放行本地预览来源

### 6.2 建议封面图习惯

建议每篇文章：

1. frontmatter 里写 `image`
2. frontmatter 里写 `heroImage`
3. 正文顶部再放一张 `<img>`

原因：

- 首页卡片用封面
- 文章正文打开后也能直接看到头图
- 排查问题更直观

### 6.3 图片不显示时怎么查

先检查：

1. 图片 URL 是否能直接打开
2. 对应 OSS bucket 是否加了本地 Referer 白名单
3. 是不是用了错误 bucket

本地预览常见白名单：

- `http://127.0.0.1:4321/`
- `http://127.0.0.1`
- `http://localhost/`
- `http://localhost:4321/`

## 7. 正文写法建议

当前这个项目对 Markdown 兼容度比较高，支持：

- 标准 Markdown
- HTML 标签混写
- 图片直链
- 代码块

建议你日常写法：

- 标题用 Markdown
- 图片尽量用标准 Markdown 或 `<img>`
- 代码块一定写语言名，但要写规范语言名

比如：

```md
```python
print("hello")
```
```

不要写错成：

- `markdwon`
- `SQL`
- `g`

因为这会在构建时触发高亮警告。

## 8. 新增文章的完整流程

### 8.1 新建文章

在 `src/pages/posts` 下新建文件，例如：

- `57-你的标题.md`

### 8.2 填写 frontmatter

至少填：

- `date`
- `issueTitle`
- `description`
- `image`
- `heroImage`

### 8.3 写正文

建议正文顶部放一张封面图。

### 8.4 本地预览

```bash
npm run dev
```

浏览器打开：

- `http://127.0.0.1:4321`

检查：

- 首页卡片是否正常
- 文章页是否正常
- 图片是否显示
- 标题是否过长
- 摘要是否顺眼

### 8.5 构建检查

```bash
npm run build
```

如果构建失败，先解决再提交。

### 8.6 提交 git

```bash
git add .
git commit -m "add post 57"
```

## 9. 修改旧文章时注意什么

如果是改旧文章，优先只改内容，不要顺手改文件名。

建议修改范围：

- 摘要
- 封面图
- 标签
- 正文错别字
- 图片失效链接

不建议随便改：

- 文件名前缀编号
- 文章整体路由逻辑

## 10. 发布前检查清单

每次发文前至少看这几项：

- 标题是否正确
- 摘要是否像人话
- 封面图是否显示
- 文章正文首图是否显示
- 首页卡片有没有裂图
- 构建是否通过
- git 提交信息是否清楚

## 11. 当前已知发文限制

- 英文版当前未上线，不要把文章直接发到 `src/pages/en/posts`
- 有少数旧文章还带着迁移时代的兜底封面策略
- 根目录有部分历史调试目录，和发文本身无关，不必碰

## 12. 最短操作路径

如果你只是想最快发一篇新文章，最短路径是：

1. 在 `src/pages/posts` 新建 Markdown
2. 填 frontmatter
3. 跑 `npm run dev`
4. 看首页和文章页
5. 跑 `npm run build`
6. git 提交

这就是日常最常用流程。
